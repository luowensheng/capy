# supercharge-markdown

**Take an existing format (Markdown) and give it superpowers.**

This sample shows how Capy can sit *on top of* an existing target —
adding structured directives, components, and validation — without
replacing the host language. The output is **plain Markdown** that any
static site generator (Hugo, Jekyll, MkDocs, Next.js, Astro) renders.

## The DSL

```
post "Adopting Capy at Acme" date "2026-05-24" author "Alice Chen"
    tag rust
    tag devtools
    tag dsl

    para "We replaced 3 hand-rolled generators with a single Capy library."

    h2 "Why Capy"
    bullet "One source file, multiple language targets — no drift."
    bullet "Library doubles as the spec."

    callout note "This post is itself generated by Capy."

    h2 "Numbers"
    card "Latency reduction" "10x" "microseconds vs. minutes"
    card "Generators retired" "3" "all replaced by 1 library"
end
```

26 lines of DSL → 48 lines of polished Markdown with YAML frontmatter,
blockquote callouts, and HTML metric cards inline.

## What it generates

- **Frontmatter block** (`---\ntitle: ...\n---`) with title, date,
  author, and tag list — exactly what every static site generator
  expects.
- **Headings, paragraphs, bullets** as normal Markdown.
- **Callouts** rendered as `> **NOTE:** ...` blockquotes — works
  everywhere.
- **Metric cards** as inline `<div class="metric-card">` HTML — your
  CSS styles them however you want.
- **Code blocks** with proper fencing and language tags.

## Why "supercharge" not "replace"

The output is still Markdown. Your existing toolchain — `hugo build`,
`jekyll serve`, `mkdocs serve`, whatever — keeps working unchanged.
Capy just runs BEFORE the static site generator:

```
script.capy ── capy run ──▶ post.md ── hugo build ──▶ public/
```

You get:

- A **typed, validated authoring surface** for content. Typos in
  `tag` names? Add a `type Tag { options ... }`. Date in the wrong
  format? `type ISODate { pattern "..." }`. Capy refuses to compile
  bad content instead of silently producing broken HTML.
- **Components** (cards, callouts, custom blocks) without learning a
  templating engine. The library defines them once; authors use them
  by name.
- **Reusable across sites.** The same `lib.capy` works for every
  blog you maintain.

## Run

```sh
../../capy run lib.capy script.capy > post.md
```

Drop `post.md` into `content/posts/` (Hugo), `_posts/` (Jekyll),
`docs/blog/` (MkDocs) — wherever your site generator looks.

## Other formats Capy can supercharge with the same pattern

| Host format       | What Capy adds                                              |
|-------------------|-------------------------------------------------------------|
| **Markdown**      | Frontmatter, callouts, cards, code-with-output (this demo)  |
| **HTML**          | Component primitives that expand to plain HTML+CSS+JS       |
| **SQL DDL**       | `pk`, `fk`, `timestamps`, audit-column macros               |
| **Dockerfile**    | `base`, `apt`, `pip`, multi-stage builds with one line each |
| **GitHub Actions**| `job ... steps ...` shorthand that expands to full YAML     |
| **Terraform HCL** | Module shortcuts, environment-aware defaults                |
| **Kubernetes**    | One-liner deployments that expand to full manifests         |

The pattern is always the same: write a Capy library whose
`file_template:` outputs the host format. The DSL becomes the
ergonomic authoring layer; the host format stays the deployment
target.
